# DatsSol — Стратегия и MVP «Червяк v1»

**Дата:** 2026-04-17
**Тип:** дизайн-документ (стратегический разбор + MVP-спецификация бота)
**Статус:** утверждён пользователем для реализации

---

## 1. Контекст

DatsSol — пошаговая стратегия, где игроки терраформируют планету, выращивая сеть плантаций от центрального юнита (ЦУ). Полные правила: `docs/specs/dats_sol_spec.md`. API: `docs/specs/openapi.yml`.

**Жёсткие константы игры (для бота):**

- 1 секунда на ход (`nextTurnIn` в ответе арены)
- 600 ходов в раунде, очки накапливаются между раундами
- Карта в координатах `[x, y]`; радиусы — квадраты Чебышева (`|Δx| ≤ R && |Δy| ≤ R`)
- Бонусные клетки: `x % 7 == 0 && y % 7 == 0` дают 1500 очков (vs 1000 обычные)
- Лимит плантаций: 30 (расширяется апгрейдом до 40)
- Очко апгрейда раз в 30 ходов, максимум 15
- Адъяцентность управления — кардинальная (без диагоналей), от ЦУ

**Порядок обработки сервера (важно для тактики):**
1. Апгрейды → 2. Стройка/Ремонт → 3. Диверсии → 4. Игрок → бобёр → 5. **Перенос ЦУ** → 6. Бобёр → игрок → 7. Деградация потерянных → 8. Урон по простаивающим стройкам → 9. Терраформинг + очки → 10. Респавн → 11. Катаклизмы

**Критическое следствие:** «достроил → переехал ЦУ» работает в один ход (стройка на шаге 2, перенос на шаге 5).

---

## 2. Анализ источников очков

| Источник | Очки | Эффективность | Используем в MVP? |
|---|---|---|---|
| Терраформинг обычной клетки (TS=5) | 50/ход | База | ✅ |
| Терраформинг бонусной клетки | 75/ход | +50% | ✅ (приоритезация) |
| Убийство бобра | 10×клетка ≈ 10–15K | Топ ROI, но требует SE-кооперации | ❌ (v2) |
| Диверсия (добивание) | очки клетки врага | Только реактивно | ❌ (v2) |

**Вывод:** для MVP единственный источник очков — терраформинг. Бонусные клетки берём оппортунистически, не через дальние маршруты.

## 3. Стратегические инварианты

### 3.1. Самоликвидация плантаций (ключевая механика)

После 100% терраформации **плантация на клетке исчезает**. Это означает:

- ЦУ нельзя оставлять на одной клетке дольше ~20 ходов (TS=5 → клетка завершится → потеря ЦУ → катастрофа)
- В линейной цепочке хвост отваливается каждые ~10 ходов (из-за разницы build-time vs terraform-time)
- Эффективное число живых плантаций в чисто-линейной стратегии = 2–3, не 30

### 3.2. Защита ЦУ — священна

Потеря ЦУ = разрушение всей сети + штраф 5% (минимум 1 очко). Правила:

- Никогда не строить около лимита плантаций (старейшая удаляется автоматически — может оказаться ЦУ)
- ЦУ должна всегда иметь смежного «преемника» готового или строящегося
- При HP ЦУ < 50% — соседняя плантация ремонтирует вместо стройки

### 3.3. Адъяцентность через кардинальные шаги

- Диагональные «острова» становятся `isIsolated` → не дают очков, деградируют
- При обрыве цепочки все плантации за разрывом теряются
- Вывод: расширяться **широким фронтом** (две параллельные линии где возможно), а не узкой нитью

### 3.4. Релейный штраф

Каждая последующая команда через ту же плантацию теряет 1 ед. эффективности (CS/RS/SE/BE), вплоть до 0.

- Несколько плантаций, бьющих в одну цель напрямую (`[author, author, target]`), штраф **не получают**
- Штраф считается только когда плантация — промежуточный узел в `path`
- В MVP полностью обходим это, не используя релеи

### 3.5. Свойства апгрейдов

| Апгрейд | Эффект | Макс | Стоит ли в MVP |
|---|---|---|---|
| `repair_power` | +1 к RS **и CS** (см. API doc) | 3 | ✅ топ-1, ускоряет стройку |
| `signal_range` | +1 к SR | (?) | ✅ для веток |
| `settlement_limit` | +1 к лимиту плантаций | 10 | ✅ для масштабирования |
| `decay_mitigation` | -2 к DS, действует на стройку | 3 | ✅ защищает строящиеся |
| `max_hp` | +10 к MHP, **не действует на стройку** | 5 | средний приоритет |
| `vision_range` | +2 к VR | 5 | низкий |
| `earthquake_mitigation` | -2 к урону, на стройку | 3 | по ситуации |
| `beaver_damage_mitigation` | -2 к урону, на стройку | 5 | по ситуации |

## 4. Полная стратегия (4 фазы)

### Фаза 0 — «Чистый червяк» (ходы 1–20)

- Одна плантация (ЦУ) → строит 1 кардинального соседа
- Все команды формата `[author, author, target]`, без релеев
- Апгрейд № 1 (ход 30) пойдёт на `repair_power` → CS 5→6 → стройка 50/6 ≈ 9 ходов

### Фаза 1 — «Червяк с подпорками» (ходы 20–60)

- 2-я плантация помогает ЦУ строить вперёд → CS=10 (без штрафа), стройка за 5 ходов
- **Задняя плантация** (та, чья клетка скоро завершится) строит **в перпендикулярном направлении** → создаёт боковую ветку, которая переживёт исчезновение хвоста
- Структура к ходу 60: «рыба» с 2–3 боковыми ветками, ~8–12 живых плантаций

### Фаза 2 — «Параллельные червяки» (ходы 60–300)

- Каждая боковая ветка — самостоятельный червяк, отращивающий собственные ветки
- Сеть = дерево; одни ветки растут, другие завершают клетки и отваливаются
- Апгрейды: `settlement_limit` для расширения лимита, `signal_range` для гибкости

### Фаза 3 — «Уплотнение» (ходы 300–550)

- Заменяем завершённые клетки новыми постройками (клетка деградирует через 80 ходов — окно для повторной застройки)
- Переход к выживанию: `decay_mitigation`, `max_hp`
- Реактивные диверсии (если вражеская клетка >70% и в радиусе)

### Фаза 4 — «Завершение» (ходы 550–600)

- Не строим около лимита — снос ЦУ катастрофа
- Все плантации только терраформируют, ремонт критичных

## 5. MVP — «Червяк v1»

**Цель:** работающий бот за ~1 час, играющий по логике Фазы 0 + старт Фазы 1. Дальше итерируем.

### 5.1. Что делает бот

1. Каждый ход: GET `/api/arena` → принять решение → POST `/api/command`
2. Стратегия: одна цепочка кардинальных шагов от ЦУ; все доступные плантации в радиусе AR помогают строить следующую клетку фронта
3. Когда фронт-плантация достроена и смежна с ЦУ → `relocateMain` в неё в том же ходу
4. Очко апгрейда → берём из захардкоженного приоритет-листа
5. Фронт направляется к ближайшему доступному узлу `(x%7==0 && y%7==0)`; если все 4 соседа заблокированы — fallback на ближайшего доступного соседа любой нашей плантации

### 5.2. Что НЕ делает MVP

- Диверсии
- Охота на бобров
- Боковые ветки (отложено в v2)
- Релеи / многошаговые `path`
- Pathfinding с обходом препятствий (только локальная фильтрация 4 соседей)
- Учёт песчаных бурь и землетрясений (пассивно — ремонтируем при низком HP)
- Кэширование/предсказание состояния

### 5.3. Архитектура

Один файл `main.py`, ~150–200 строк. Структура:

```
config        — токен, base_url из .env
http_client   — httpx.Client, get_arena() и post_command()
models        — pydantic BaseModel: Arena, Plantation, Beaver, Construction, Mountain, Cell, Meteo
brain
  ├─ pick_target(arena)        → координата следующей клетки фронта
  ├─ build_commands(arena, t)  → список path-команд
  ├─ check_relocate(arena)     → (bool, [from, to]) для relocateMain
  └─ pick_upgrade(arena)       → имя апгрейда или ""
loop          — while True: arena → decide → command → sleep(nextTurnIn)
```

### 5.4. Стек библиотек

| Библиотека | Назначение | Почему |
|---|---|---|
| `httpx` | Sync HTTP-клиент | `Client(base_url=..., headers={X-Auth-Token: ...}, timeout=...)` — компактно и явно |
| `pydantic` v2 | Парсинг JSON-ответов арены в типизированные модели | Поле `validation_alias` маппит camelCase API → snake_case Python; ошибки валидации сразу видны |
| `tenacity` | Retry на сетевых ошибках | `@retry(stop=stop_after_attempt(2), wait=wait_fixed(0.1), retry=retry_if_exception_type(httpx.HTTPError))` — чтобы не падать из-за одного дропа |
| `python-dotenv` | Загрузка `.env` | Токен в `.env`, не хардкод |
| `loguru` | Логирование | Из коробки красивый вывод, без boilerplate `logging.basicConfig` |

Установка одной командой: `venv/bin/pip install httpx pydantic tenacity python-dotenv loguru`.

### 5.5. Логика принятия решения (псевдокод)

```python
def decide_turn(arena: Arena) -> Command:
    hq = next(p for p in arena.plantations if p.is_main)
    controllable = [p for p in arena.plantations if not p.is_isolated]

    # 1. Выбор цели фронта
    target = pick_target(arena, hq)
    # — кандидаты: 4 кардинальных соседа HQ (N/E/S/W)
    # — фильтр: не гора, не занята своими/чужими/стройкой, не в радиусе 2 от бобра
    # — скоринг: для каждого кандидата считаем манхэттен-расстояние до ближайшего
    #   бонусного узла (x%7==0 && y%7==0); выбираем кандидата с минимальным расстоянием
    # — tie-break: направление, стабильное от хода к ходу (сохраняем last_direction в state)

    # 2. Команды стройки от всех плантаций в радиусе AR от target
    AR = arena.action_range
    builders = [p for p in controllable
                if abs(p.position.x - target.x) <= AR
                and abs(p.position.y - target.y) <= AR]
    commands = [Command(path=[p.position, p.position, target]) for p in builders]

    # 3. Перенос ЦУ: если есть только что достроенная плантация, смежная с HQ
    # «Только что достроенная» = плантация с immunityUntilTurn == turn_no + 3
    # (3 хода неуязвимости даются сразу после завершения стройки)
    relocate = None
    fresh_neighbors = [
        p for p in controllable
        if is_cardinal_neighbor(p.position, hq.position)
        and p.immunity_until_turn - arena.turn_no >= 2  # построена в этот ход
        and not p.is_main
    ]
    if fresh_neighbors:
        target_hq = fresh_neighbors[0]
        relocate = [hq.position, target_hq.position]

    # 4. Апгрейд
    upgrade = ""
    if arena.plantation_upgrades.points >= 1:
        upgrade = pick_next_upgrade(arena.plantation_upgrades.tiers)

    # 5. Защита от пустого хода
    if not commands and not relocate and not upgrade:
        # экстренный fallback: попытка построить в любого свободного соседа HQ
        commands = emergency_build(arena, hq)

    return Command(commands=commands, relocate_main=relocate, plantation_upgrade=upgrade)
```

### 5.7. Защита от факапов

| Риск | Защита |
|---|---|
| Пустая команда → ошибка сервера | Если нет действий → отправляем хотя бы `plantationUpgrade` или экстренную стройку |
| HQ-клетка >80% и нет смежной замены | Стартуем стройку во ВСЕ свободные кардинальные соседи HQ одновременно |
| Сетевой timeout | `tenacity` retry × 2 с малой задержкой; при полном фейле — пропускаем ход (без crash) |
| `nextTurnIn` пропущен | `sleep(1.0)` как fallback |
| Превышение лимита плантаций | Никогда не строим, если `len(plantations) >= settlement_limit_current - 1` |
| Бобёр в радиусе 2 от target | Target отбрасывается, fallback на другого соседа |
| Незарегистрированный токен | Падаем с понятным сообщением при старте |

- [ ] Бот стартует и не падает на пустом ответе арены
- [ ] Цикл крутится 100+ ходов подряд без crash

- [ ] Применён минимум 1 апгрейд
- [ ] Очки тикают на дашборде `https://gamethon.datsteam.dev/datssol/stats`

### 5.10. Готовность к итерациям (v2+)

После того как v1 работает, добавляем по очереди:

1. **Боковые ветки** — задняя плантация вместо помощи фронту строит вбок
2. **Защита от бурь** — если буря пройдёт через нашу клетку в ближайшие 5 ходов, ремонтируем заранее
3. **Граф управления через `networkx`** — точное определение `isIsolated` на стороне бота, не доверяя только серверному флагу
4. **Охота на бобров** — групповой удар когда 4+ плантаций в радиусе AR
5. **Реактивные диверсии** — добиваем вражеские клетки на 70%+
6. **Async HTTP** — `httpx.AsyncClient` для параллельного фетча `/arena` и `/logs`

## 7. Открытые вопросы для уточнения

- Точное значение `max` у `signal_range` в `tiers` — спека не фиксирует, узнаем из ответа арены при первом запуске
- Поведение `relocateMain`, если целевая плантация ещё не достроена в этот ход (только что прошла шаг стройки) — проверить эмпирически
- Реальное время round-trip к серверу (для уточнения бюджета) — измерим в первом же запуске

Эти вопросы не блокируют MVP — задаём дефолты и корректируем после первых раундов.
